# 安全策略

感谢你关注本仓库的安全。我们重视负责任的安全披露，并会认真对待每一份有效的漏洞报告。

## 适用范围

本策略适用于本仓库及其在默认分支上公开分发的内容，包括但不限于：

- 仓库中的源代码、脚本与配置文件
- 文档、示例与自动化工作流（如 GitHub Actions）
- 通过本仓库 README 或文档链接引用的、由维护者明确托管的资源

以下情况通常**不在**本策略范围内（除非你确信问题直接源于本仓库的代码或配置）：

- 第三方服务、依赖包或上游项目的已知问题（请先向其维护者报告）
- 仅影响你本地 fork 或未合并分支的修改
- 社会工程、钓鱼、垃圾信息等非技术类威胁
- 对公开信息（如已公开的邮箱、用户名）的枚举，且未导致实质性安全影响

## 支持的版本

安全修复优先应用于**当前默认分支**上的最新内容。若本仓库发布带版本号的发行版（Release），我们原则上仅对**尚未结束生命周期的最新主要版本**提供安全更新；更旧版本可能不再维护。

| 范围 | 是否接受安全报告 |
| ---- | ---------------- |
| 默认分支（`main` / `master` 等）最新提交 | ✅ 是 |
| 已发布且未标注停止维护的 Release | ✅ 是（视情况回溯修复） |
| 已归档、停止维护或明确弃用的分支/版本 | ❌ 否 |
| 仅存在于未合并 Pull Request 中的代码 | ⚠️ 视具体情况评估 |

若仓库未使用语义化版本或 Release，请以**默认分支最新状态**为准。

## 如何报告漏洞

**请勿**通过公开的 GitHub Issue 讨论可被利用的安全漏洞，以免在未修复前扩大暴露面。

请通过以下**私密渠道**之一报告：

1. **GitHub 私密安全通告（推荐）**  
   在仓库页面选择 **Security → Advisories → Report a vulnerability**（若已启用）。
2. **电子邮件**  
   发送至维护者在 GitHub 个人资料或本仓库 README 中列出的联系邮箱；邮件主题建议包含 `[SECURITY]` 前缀。

报告时请尽量提供以下信息，以便我们更快复现与修复：

- 问题类型与受影响组件（文件路径、功能、工作流名称等）
- 复现步骤或概念验证（PoC），以及环境说明（分支、提交哈希、运行环境）
- 潜在影响评估（机密性、完整性、可用性）
- 你是否认为该问题已被公开，以及是否愿意参与协调披露

## 响应与披露时间线

我们会在收到报告后尽快确认收悉，并尽力按以下节奏处理（具体取决于严重程度与复现难度）：

| 阶段 | 目标时间（工作日） |
| ---- | ------------------ |
| 初步确认与分类 | 约 3 个工作日内 |
| 严重程度评估与修复计划沟通 | 约 7 个工作日内 |
| 修复发布或临时缓解措施 | 视严重程度，通常 30 天内（严重问题优先） |

**你可能收到的情况：**

- **已确认**：我们会在修复就绪后与你沟通；若你同意，可在修复发布后于安全通告或致谢中提及（可选）。
- **无法复现或信息不足**：我们会说明需要补充的材料，并在合理期限内等待你的回复。
- **不在范围内或误报**：我们会解释原因；若属于上游或第三方问题，会尽可能指出更合适的报告渠道。
- **已知问题或重复报告**：我们会告知现有跟踪状态，避免重复劳动。

我们遵循**协调披露**原则：在修复可用前，请勿公开漏洞细节；修复发布后，我们欢迎与你共同确定合理的公开说明时间。

## 安全研究与测试须知

我们欢迎出于改进安全目的的研究，但请遵守：

- 仅对你有权测试的环境进行操作；不得访问、修改或删除他人数据。
- 避免对生产或公共服务造成拒绝服务（DoS）、大规模扫描或自动化爆破。
- 不得利用漏洞索取报酬以外的额外利益，或从事任何违法活动。

本策略不构成对任何司法管辖区法律的豁免；研究者须自行遵守适用法律与平台服务条款。

## 致谢

我们感谢负责任地披露安全问题的研究人员与用户。经报告者同意，我们可能在安全通告、Release 说明或本文件的更新记录中予以致谢。
