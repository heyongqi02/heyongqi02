# 贡献指南

感谢你愿意为本项目做出贡献。无论是修复缺陷、改进文档、提出想法，还是提交代码，我们都非常欢迎。

## 行为准则

参与本仓库相关讨论与协作时，请遵守 [行为准则](./CODE_OF_CONDUCT.md)。我们致力于为所有人提供友好、包容的环境。

## 你可以如何贡献

| 方式 | 说明 |
| ---- | ---- |
| **报告问题** | 通过本仓库的 **Issues** 描述缺陷、改进建议或文档问题；提交前请先搜索是否已有同类讨论。 |
| **提交代码** | 通过 Pull Request 修复 bug、实现功能或优化实现；较大改动建议先开 Issue 与维护者对齐方向。 |
| **改进文档** | README、注释、示例与本文档的修正与补充同样有价值。 |
| **安全披露** | 请勿在公开 Issue 中讨论安全漏洞，请按 [安全策略](./SECURITY.md) 进行负责任披露。 |

## 开发环境

### 前置要求

请根据项目根目录中的配置文件准备环境，常见包括：

- 版本控制系统（如 [Git](https://git-scm.com/)）
- 运行时与包管理器（版本要求见 `package.json` 的 `engines`、`packageManager`，或 `.node-version`、`.nvmrc` 等文件）

具体工具与版本以仓库说明为准。

### 克隆与安装

```bash
git clone <仓库 URL>
cd <项目目录>
# 安装依赖 — 命令见 README 或 package.json
```

## 本地开发

构建、测试、静态检查等命令以 **README** 与 **`package.json` 的 `scripts`** 为准。提交 PR 前，请完成维护者在 PR 模板或 CI 中要求的本地检查（通常至少包括测试与 Lint/类型检查）。

若仓库配置了持续集成，在默认分支上的 push 与 Pull Request 一般会触发自动化检查；本地先跑通相同流程，可减少审查往返。

## 提交 Pull Request

1. **Fork** 本仓库，并基于默认分支创建功能分支（命名建议：`fix/...`、`feat/...`、`docs/...`）。
2. **聚焦改动**：一个 PR 尽量只做一件事，便于审查与回溯。
3. **填写 PR 说明**：使用仓库提供的 [PR 模板](./.github/PULL_REQUEST_TEMPLATE.md)（若有），说明解决了什么问题；若关联 Issue，可使用 `fixes #123` 等关键字。
4. **补充测试**：若改动涉及逻辑，请尽量添加或更新测试，确保合并前后行为可验证。
5. **避免重复**：确认是否已有 PR 以相同方式解决同一问题。

### 关于 AI 辅助编写

我们理解 AI 工具可提高效率，但**不接受**未经人工逐行审查的「Vibe coding」或未审阅的 AI 生成核心代码（文档类改动除外）。若 PR 包含 AI 生成内容，请在 PR 中勾选相应项并说明你已经自行仔细审查。

## 报告 Issue

提交 Issue 时请尽量包含：

- **环境**：操作系统、运行时与依赖版本
- **复现步骤**：从干净安装到问题出现的完整路径
- **期望与实际**：你期望发生什么，实际发生了什么
- **补充信息**：日志、截图或最小复现示例

若仓库提供了 Issue 模板，请优先使用以加快处理。

## 版本与发布

版本号与发布流程由维护者管理；贡献者通常无需关心发布细节。若你的改动会影响对外 API 或公开接口行为，请在 PR 中明确说明，便于维护者评估是否需要变更日志或主版本号。

## 许可

你提交的贡献将按本仓库 [LICENSE](./LICENSE) 授权。提交 PR 即表示你同意在相同许可下授权你的贡献。
